# February 26, 2026

## First Open Source Contribution to Reth

Today I submitted my first PR to Reth. Found that the transaction pool's ordering logic was implemented differently from Geth and created a PR to fix it.

### The Problem

While looking at Reth's `CoinbaseTipOrdering`, I noticed a comment: "NOTE: The implementation is incomplete for missing base fee". The code was only comparing `effective_tip_per_gas`.

Looking at Geth's implementation, it compares in 3 levels:

```go
func (h *priceHeap) cmp(a, b *types.Transaction) int {
    // 1. Compare effective tip
    if h.baseFee != nil {
        if c := a.EffectiveGasTipCmp(b, h.baseFee); c != 0 {
            return c
        }
    }
    // 2. Compare fee cap
    if c := a.GasFeeCapCmp(b); c != 0 {
        return c
    }
    // 3. Compare tip cap (final tie-breaker)
    return a.GasTipCapCmp(b)
}
```

### The Solution

I was wondering how to compare multiple values sequentially in Rust - turns out tuples do exactly that.

Rust tuples automatically provide lexicographic ordering:

```rust
(1, 500, 100) > (1, 400, 200) // compares second element
(1, 500, 100) > (1, 500, 50) // compares third element
```

So I changed `PriorityValue` from `u128` to `(u128, u128, u128)`:

```rust
type PriorityValue = (u128, u128, u128);

fn priority(...) -> Priority {
    let effective_tip = transaction.effective_tip_per_gas(base_fee).unwrap_or(0);
    let fee_cap = transaction.max_fee_per_gas();
    let tip_cap = transaction.max_priority_fee_per_gas().unwrap_or(0);

    Priority::Value((effective_tip, fee_cap, tip_cap))
}
```

This automatically implements Geth's 3-level comparison logic. Nice example of leveraging language features for cleaner code.

### Test Updates
